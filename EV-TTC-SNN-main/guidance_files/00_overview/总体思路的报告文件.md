# code_audit_for_m3ed_ttc

## 0. 审计结论

本报告按 `guidance_files/event_flow_transfer_EVTTC总体思路.md` 要求完成代码审计，当前不开始实现。

核心判断：

- MAVLab `event_flow-main` 已经具备目标训练节奏：连续 `1000 events` 一步，累计 `10000 events` 后做一次 backward，随后 `detach_states()`，序列切换时 `reset_states()`。

  【detach_states()作用：保留网络当前的数值，但切断这些状态与此前计算图之间的梯度连接，否则膜电位$$V_t$$会一直连接$$V_{t-1}、V_{t-2}...V_{t-n}$$，反响传播的链路越来越大】

- MAVLab 的 SNN/ANN 网络结构、LIF/PLIF/ALIF/XLIF 神经元、替代梯度、状态管理可以复用；但原来的 `EventWarping` 自监督光流 loss 不能直接用于 TTC，需要改成监督式 TTC loss。

- EV-TTC 当前 TTCEF 流程是按约 `7 ms` readout 生成 `exp_filts`，再用 `calc_gt.py` 按 `end_time - dt -> end_time` 生成 `depth/TTC/flow/mask`，最后 `merge.py` 做速度、角速度、event、TTC 上限等过滤。

- 当前 EV-TTC merged H5 是随机单帧训练格式，不适合直接训练“1k events × 10步 BPTT”的 SNN TTC；需要新写 M3ED 连续事件 dataloader 或先生成连续 1k-step TTC H5。

- `calc_gt.py` 中 TTC 标签对应 start/end 时刻都有信息：`ts=start_times`、`te=end_times`。当前 TTC 使用 start 时刻深度图、start->end 相对运动，公式为 `ttc = depth / (Tz + 1e-5)`。

- 当前 mask 由多阶段组成：GT 深度有效、时间边界有效、速度/角速度过滤、event 活跃、TTC finite、TTC < 100。负 TTC 没有被过滤。

建议下一阶段先做最小实现：

1. 新建 `M3EDTTCEventLoader`，直接从 M3ED raw `prophesee/left/x/y/t/p` 切 1k events。
2. 每个 1k step 对齐一个 start 或 end 标签索引，返回 `event_cnt/event_list/ttc/depth/mask/supervise_valid`。
3. 复用 `LIFFireNet`/`SpikingRecEVFlowNet` 主体，改最后输出为 `1` 通道 TTC 或 inverse TTC。
4. 新写 `TTCBPTTLossBuffer`，累计 10 步中有效 step 的 masked Charbonnier loss，统一归一化后 backward。

## 5. 时间组织方式

### 5.1 event_flow 当前训练时间组织

- 每次 `H5Loader.__getitem__` 读取 `config["data"]["window"]` 个事件。`train_SNN.yml` 中为 `1000`。
- `EventWarping.event_flow_association()` 每次 forward 后把当前 1k 的事件、mask、flow 拼到内部 buffer。
- `loss_function.num_events >= config["data"]["window_loss"]` 时触发 backward。`train_SNN.yml` 中 `window_loss=10000`，等价 10 个 step。
- backward 后：
  - `optimizer.step()`
  - `optimizer.zero_grad()`
  - `model.detach_states()`
  - `loss_function.reset()`
- 序列切换或事件不足：
  - `data.new_seq = True`
  - 外层训练循环执行 `loss_function.reset()`、`model.reset_states()`、`optimizer.zero_grad()`
  - 不足 10 步的 buffer 不会 backward，直接丢弃。

关键源码：

```python
# event_flow-main/dataloader/h5.py
if self.config["data"]["mode"] == "events":
    event_idx0 = self.batch_row[batch]
    event_idx1 = self.batch_row[batch] + window
...
self.batch_row[batch] += self.config["data"]["window"]
```

```python
# event_flow-main/train_flow.py
if data.new_seq:
    loss_function.reset()
    model.reset_states()
    optimizer.zero_grad()
```

### 5.2 EV-TTC 当前时间组织

- `create_exp.py` 不是固定事件数，而是按事件时间流生成 readout 帧。
- 默认配置中 `output_time=7 ms`，因此大约每 7ms 保存一次 `exp_filts`。
- `calc_gt.py` 对每个 `exp_time` 生成一个标签：
  - `te = exp_time`
  - `ts = te - dt`
  - 默认 `dt=0.010 sec`
- 当前 merged H5 保留的是经过过滤后的单帧样本，不保留每个样本的连续状态。

关键源码：

```python
# EV-TTC-main/TTCEF/create_exp.py

    last_active = t_ev[i]
    exp_img = exp_img * filt_constants[:, -1][:, None, None]
    exp_times[cnt] = last_active
    save_imgs[cnt] = exp_img
```

### 5.3 迁移目标需要的时间组织

目标与 event_flow 一致：

```text
1k events step 0 -> forward -> 可选择监督/不监督
1k events step 1 -> forward -> 可选择监督/不监督
...
1k events step 9 -> forward -> 计算所有有效 step/pixel loss -> backward
```

关键差异：

- 事件分片必须来自 M3ED raw event stream，而不是 EV-TTC 的 7ms `exp_filts`。
- 每个 1k step 的标签应由该 step 的 start/end timestamp 对齐生成。
- 对无效速度、角速度、标签或 mask 的 step，仍 forward 更新 SNN 状态，但不参与 loss。
- 序列边界、不足 10 step、时间跳变处必须 reset buffer 和 SNN state。

## 6. 空间处理

### 6.1 去畸变、裁剪、下采样

`create_exp.py`：

- 读取 Prophesee left camera intrinsics/distortion。
- `cv2.getOptimalNewCameraMatrix` + `cv2.initUndistortRectifyMap` 得到 undistort map。
- 只使用原图中央 `x in [280,1000)`，`y in [0,720)`。
- 通过 `calc_dist_map_ds()` 映射到 `360x360`，并做四邻域能量分配。

源码：

```python

...

```

`calc_gt.py`：

- 使用同一个 undistort camera matrix。
- depth 点先投影到 undistorted frame，再做相同 crop/downsample。

源码：


### 6.2 event_flow 增强策略

- `BaseDataLoader` 为每个 batch lane 保存 `batch_augmentation`。
- `reset_sequence()` 时重新采样增强参数。
- 同一连续序列内增强参数保持不变，直到 sequence reset。
- 原项目只保证 events/frames/flowmap 同步；TTC/depth/mask 需要新写同步增强。

源码：

```python
# event_flow-main/dataloader/base.py
self.batch_augmentation[mechanism] = [False for i in range(batch_size)]
...
def reset_sequence(self, batch):
    for i, mechanism in enumerate(self.config["loader"]["augment"]):
        self.batch_augmentation[mechanism][batch] = np.random.random() < prob
```

迁移注意：

- TTC、depth、mask 是标量图，水平/垂直翻转不改变数值符号。
- flow 若作为辅助输出/可视化，则水平翻转要改 `flow_x` 符号，垂直翻转要改 `flow_y` 符号，event_flow 已有 `augment_flowmap()` 可参考。
- 若使用 crop，则事件坐标和标签图必须严格使用同一 crop 参数；连续 10 步内增强/crop 参数应固定。

## 7. TTC 标签生成细节

### 7.1 标签对应 start_time 还是 end_time

当前 EV-TTC `calc_gt.py` 同时保存：

```python
f_w.create_dataset("ts", data=start_times)
f_w.create_dataset("te", data=end_times)
```

TTC/depth/flow 的生成逻辑：

- `end_times = exp_times`
- `start_times = end_times - dt`
- depth 使用 `start_times` 附近的 depth frame，并重投影到 start event camera frame。
- 相对运动使用 start pose 和 end pose。
- flow 是把 start frame 的点变换到 end frame后投影差。

因此当前标签本质是：**start frame 几何状态 + start 到 end 的相对运动，对应这个时间间隔的 TTC/flow**。

### 7.2 深度重投影到哪个时刻

源码：

```python
# EV-TTC-main/TTCEF/calc_gt.py
start_depth = np.searchsorted(depth_ts, start_times)
depth_raw = depth[int(start_depth[idx])]
Dn_T_C0 = depth_pose[start_depth[idx]]
CN_T_DN = CN_T_C0[idx] @ np.linalg.inv(Dn_T_C0)
event_cord_N = CN_T_DN @ depth_cord_N_aug
```

结论：

- depth 选择 `start_times` 之后的最近 depth index。
- depth pose 被变换到 `CN`，也就是 start event camera frame。
- 不是对 depth 做时间插值，而是 `searchsorted` 取最近后一帧。

### 7.3 T/Omega 坐标系和 Tz 符号

源码：

```python
CN_T_CNP1 = CN_T_C0 @ inv(CNP1_T_C0)
T = CN_T_CNP1[:, :3, -1] / dt
Omega = logm(CN_T_CNP1[:, :3, :3]) / dt
ttc = depth_img / (T[idx, 2] + 1e-5)
```

按变量命名，`CN_T_CNP1` 表示 end camera 到 start camera 的相对变换，`T` 被表达在 start camera frame 下。当前代码直接使用 `Tz` 作为接近/远离的轴向速度。

风险：

- `Tz` 的正负约定需要和 M3ED pose convention、相机坐标 z 轴方向一起再做一次数值确认。
- 当前实际数据中存在大量负 TTC，说明代码不会把远离场景过滤掉。

### 7.4 负 TTC、低速和最大 TTC

当前处理：

- 负 TTC：`merge.py` 没有过滤，因为只做 `ttc_chunk < 100`，负值满足条件。
- 低速：按 `||T||` 过滤，不是按 `Tz` 过滤。
  - car: `||T|| > 1.3`
  - non-car: `||T|| > 0.25`
- 大角速度：`||Omega|| < 0.18`
- 最大 TTC：`ttc < 100`。

这意味着：

- 侧向运动但 `||T||` 足够大也可能保留。
- 远离运动的负 TTC 也可能保留。
- 若训练目标只关注 approaching TTC，需要新增 `ttc > 0` 或 inverse TTC 风险标签策略。

### 7.5 mask 最终构成

当前 GT mask 初始为：

```python
# EV-TTC-main/TTCEF/calc_gt.py
mask = ~np.isnan(depth_img)
```

merge 后最终 mask：

```text
final_mask =
  valid_exp_time
  & speed/omega sample-level valid
  & finite depth/ttc pixel
  & event active at last exp channel
  & ttc < 100
```

源码：

```python
mask_chunk = mask_chunk[valid_mask] & (np.abs(exp_chunk[:, -1]) > 1e-3) & ~np.isnan(ttc_chunk)
ttc_chunk = np.nan_to_num(ttc_chunk)
mask_chunk = mask_chunk & (ttc_chunk < 100)
```

### 8.4 当前风险或不确定部分

1. `calc_gt.py` 的 `valid_exp_mask` 只用于写空图，`start_times/end_times` 仍对全部 exp_times 生成；迁移到 1k step 时应显式保存 `supervise_valid`，不要依赖全零标签。
2. `Tz` 符号必须确认；当前公式允许负 TTC。
3. `depth` 现在用 `searchsorted` 而非插值；1k step 的时间戳更密，是否需要 depth 插值要评估。
4. event_flow 的默认输入 H5 schema 与 M3ED schema 不同，不能直接用 `H5Loader`。
5. event_flow 的 `event_formatting` 会把每个 1k partition 的 ts 归一化到 `[0,1]`，这适合 event warping；TTC 监督是否需要保留真实 `dt` 应另存 metadata。
6. 旧 EV-TTC `TTCEF_DM` 使用随机单帧 shuffle，不适合 SNN stateful 训练。
7. 当前 EV-TTC augmentation 有 `RandomRotation(0,180)`，对 TTC/depth/mask 空间上同步，但对事件 raw 坐标连续流不直接适用；迁移时建议先只做 H/V flip。
8. `event_flow` `__len__` 返回固定 `1000`，不是实际样本数；迁移 dataloader 应实现真实长度或显式 epoch policy。
9. `event_flow` 序列末尾不足 10 步会丢弃 loss；若 TTC 数据稀疏，要统计被丢弃比例。

## 10. 下一步建议

不要直接在原 `event_flow-main` 或 `EV-TTC-main` 上大改。建议在 `EV-TTC-SNN-main` 下新建独立迁移代码，逐步复用模块：

```text
EV-TTC-SNN-main/
  configs/
    train_m3ed_ttc_lif_firenet.yml
  snn_ttc/
    data/m3ed_ttc_event_loader.py
    models/lif_firenet_ttc.py
    losses/ttc_bptt_loss.py
    train_ttc_snn.py
    tools/make_1k_ttc_debug_set.py
    tools/visualize_1k_ttc_alignment.py
```

推荐最小验证顺序：

1. 选 1 个短序列，生成 64 个连续 1k step 的 debug H5。
2. 可视化每个 step 的 event count、TTC、inverse TTC、depth、mask、valid flag。
3. 用 ANN/FireNet-TTC 做 32 step overfit。
4. 用 LIF-FireNet-TTC 做 32 step overfit，确认 10-step BPTT loss 能下降。
5. 再扩展到 flight/non-flight 的小规模 train/test。
